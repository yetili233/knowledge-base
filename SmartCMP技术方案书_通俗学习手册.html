SmartCMP 技术方案书
通俗学习手册

基于《CloudChef CMP Technical Solution Proposal》（英文，11 章 162 页）整理。每章用大白话讲清"这章在卖什么能力"，附 PM 视角提醒和中英术语对照。

一句话读懂整本方案书：SmartCMP 把企业散落在各家云上的资源统一管起来——用组件和蓝图定义"能交付什么"（第2章），对接 20+ 种云把资源接进来（第3章），让用户自助申请、自动审批、自动部署（第4章），再配上 CI/CD 流水线（第5章）、算账分钱（第6章）、安全合规检查（第7章）、对接企业已有系统（第8章），最后讲平台自身的定制能力（第9章）和高可用部署规格（第10章）。
第1章方案总览平台是什么、由哪7块能力组成
大白话：这章是"开场白"——企业又有私有云又有公有云还有容器，各管各的太乱，SmartCMP 就是把它们装进一个统一控制台的平台。

7 块核心能力　①云对接 ②IaC 编排 ③自助服务 ④多租户管理 ⑤统一监控告警 ⑥FinOps 成本管理 ⑦开放集成。这 7 块正好对应后面各章——记住这个目录，整本书的骨架就有了。

关键主张　"cloud agnostic（云无关）"——用户不用关心资源跑在哪朵云上，平台按策略自动选。这是整本方案反复强调的卖点。

PM 提醒：给客户做汇报时，这 7 块能力就是你的标准话术框架。客户问"你们平台能干嘛"，按这 7 条说，不会漏。
第2章编排引擎（IaC）组件 + 蓝图 = 平台的发动机
大白话：组件像乐高积木（一台虚机、一个数据库都是一块积木），蓝图就是拼装图纸（三台虚机 + MySQL 主从拼成一套环境）。客户要什么环境，画好图纸，以后一键复制。

组件分 4 类　IaaS 组件（虚机/网络/存储）、PaaS 组件（云上的 RDS、对象存储）、软件组件（MySQL、Redis 等装在机器里的软件）、集成组件（把 ITSM、备份系统也包装成"积木"参与编排）。

蓝图不绑定云　设计图纸时不指定部署到哪朵云，用户申请时平台按资源池策略自动匹配——这就是"云无关"的具体实现。

兼容主流编排器　能调用 Terraform、Helm、Heat——客户已有的 TF 脚本和 Helm Chart 可以直接复用，号称"编排器的编排器"。

集群弹性伸缩　Day0 设计蓝图 → Day1 一键建集群 → Day2 按策略自动扩容/缩容（手动、定时、告警触发三种方式）。

PM 提醒：客户问"我们已经写了一堆 Terraform，迁过来要重写吗？"——答案是不用，平台可以直接调 TF 脚本。这是打消客户顾虑的关键话术。
第3章多云对接支持20+种云，篇幅最长但套路统一
大白话：这章像一本"插座兼容清单"——华为、阿里、AWS、Azure、VMware、K8s……每种云一小节，格式都一样：怎么连（填账号密码/AK SK）+ 支持哪些资源（一张功能清单）。不用背，用到哪查哪。

资源池三件套　配额管控（CPU/内存/存储/虚机数都能限）、超分比例（物理资源可以"卖超"，提高利用率）、标签智能调度（给池打标签如 env:production，申请时按标签自动匹配最优池）。

Cloud Gateway 云网关　客户的网络隔离区（如 DMZ、专属 VPC）平台够不着怎么办？部署一个轻量网关做"中转站"，不用对外开端口。多个网关按标签匹配资源池。这是金融类客户必问的能力。

跨云 IP 池　多朵云用同一网段容易 IP 冲突，平台统一管 IP 的规划、分配、回收和冲突检测。

华为系覆盖最全　DCS、FusionCompute、华为云、Huawei Cloud Stack 四种都支持——这和你们 GSB-TKC 项目的华为技术栈直接相关。

PM 提醒：售前阶段客户最常问"你们支持 XX 云吗，支持到什么程度"。答案都在各小节的 function list 里——回答前先查清单，不要凭印象承诺。
第4章多租户与自助服务全书最厚，平台的日常使用面
大白话：这章讲"人怎么用这个平台"——部门是租户、互相隔离，用户在服务目录里像逛淘宝一样下单，审批流自动跑，资源自动交付，之后的运维监控也在这一章。

租户两道闸　资源配额（最多用多少 CPU/内存）+ 预算配额（最多花多少钱，花完可以"禁止新申请"或"自动关机"）。一道管量、一道管钱。

工作流 4 种模式　串行、并行（HR 和 IT 同时审）、条件分支（金额大走严格流程）、状态机（工单从"打开"到"处理中"到"解决"）。拖拽设计，不用写代码。

工单服务带 AI　服务团队 + 知识库 + LLM 聊天机器人 + SLA 管理——平台自带轻量 ITSM，还有 AI 客服回答常见问题。

运维与监控　应用栈整体启停、虚机批量操作、租期回收（lease 到期关机 → 保留期过后删除）、健康热力图、告警自愈——和初始化配置手册第 7-8 章是同一套东西的售前版描述。

PM 提醒：这章和你之前学的《初始化配置手册》第 2、4、6、7、8 章高度重叠——方案书讲"有什么能力"，配置手册讲"怎么配出来"。两份对照看，售前话术和落地操作就都有了。
第5章DevOps 与 DevSecOps流水线：从发代码到上线全自动
大白话：开发提交代码 → 自动构建 → 自动测试 → 自动部署，整条流水线在平台里可视化拖拽搭出来。DevSecOps 就是在流水线里加几道"安检门"——代码扫描、漏洞检查不过关就不让上线。

流水线能力　可视化低代码设计、串行/并行/条件执行、四种触发方式（手动、定时、Webhook、Jenkins 触发）、统一看板监控执行状态。

安全左移　SAST（扫源码）、SCA（扫开源依赖）、DAST（扫运行中应用）都可以嵌进流水线；上生产前可以加"安全团队人工审批"这道闸；密钥通过 Vault 等 PAM 系统在运行时取，不落在代码里。

PM 提醒：这是个加分项而非必选项——大多数 CMP 项目首期不含 DevOps 范围。客户如果聊到这章，先确认是否在 SOW 里，别让"顺便做个流水线"变成隐形范围蔓延。
第6章FinOps 计量计费算清楚每一分钱花在哪
大白话：私有云本来没有"账单"，这章给它定价（CPU 每核每小时多少钱），公有云的真实账单也同步进来，最后按部门/项目分摊，谁花了多少一目了然——甚至能让租户像手机话费一样充值付费。

三步算账　①定价（私有云按单价模型，公有云同步真实账单）→ ②分摊（多级成本中心，按策略分到部门）→ ③呈现（明细、报表、大屏，支持成本校准和归属调整）。

成本优化建议　平台内置优化策略（闲置虚机、僵尸存储、漂浮 IP），给出修复建议并支持一键修复。

真金白银模式　支持预付费（先充值后消费）和后付费（月底出账单结算），可对接支付网关在线充值，支持多币种和税率配置——这已经是"内部云服务商"的玩法了。

PM 提醒：注意区分客户要的是 showback（只展示，不真扣钱）还是 chargeback（真结算）。后者涉及财务流程和支付对接，复杂度完全不同——需求调研时必须钉死这一点。
第7章安全合规治理自动巡检 + 一键修复
大白话：平台像个不知疲倦的安全巡检员——持续扫描有没有违规配置（比如对公网敞开的存储桶、没打补丁的虚机），发现问题告警，能修的一键修，整个过程留痕可审计。

五个环节　策略管理（JS 自定义规则）→ 自动扫描 → 实时监控告警 → 修复工作流（自动或引导式）→ 审计报告。内置虚机/RDS/安全组/存储的合规检查策略。

策略作用域灵活　可以按租户、按 Owner、按云账户、按资源池分别设不同的合规标准。

PM 提醒：方案提到 GDPR、HIPAA、PCI-DSS——但国内项目客户真正关心的是等保。售前时把平台的密码策略、MFA、审计日志、IP 黑白名单能力对着等保条款讲，比念国际标准名词管用。
第8章第三方系统集成和客户已有的IT系统打通
大白话：企业里早就有一堆系统——ITSM 工单、堡垒机、备份、CMDB、监控……CMP 不是要取代它们，而是把它们包装成"集成组件"接进编排流程，形成自动化闭环。比如：申请虚机 → ServiceNow 自动开工单 → 部署完自动注册到堡垒机和备份系统。

覆盖的系统类型　ITSM/ServiceNow、第三方监控和日志、备份容灾、堡垒机、安全平台、IPAM、终端管理、Git 版本控制——共 10 类。

集成原理统一　都走"可扩展集成框架"：把第三方系统封装成服务集成组件，暴露操作和属性，然后就能在蓝图和工作流里编排——和第 2 章的组件机制是同一套。

PM 提醒：每一项集成都是实打实的工作量。客户说"顺便和我们的堡垒机对接一下"时，要按集成开发评估工时和报价，写进 SOW，不要当成配置项白送。
第9章平台自身功能换皮、加锁、发通知、看报表
大白话：这章是平台的"周边配套"：界面换成客户的 Logo 和配色（白标）、登录加 MFA 双因素、操作全程留审计日志、通知走邮件/短信/钉钉/飞书、内置资源和容量分析报表，还有完整的 API/CLI 给客户自己写自动化。

多语言　界面支持多语言切换——国际项目（比如 OCBC 这类）的硬需求。

容量分析　资源池容量趋势分析，提前预警"池快满了"——给管理层看的运营数据。

PM 提醒：UI 白标是性价比之王——半天工作量，客户感知极强。项目早期就做掉，之后每次演示客户看到的都是"自己的平台"。
第10章部署架构平台自己怎么装、要多少机器
大白话：前面都在讲平台管别人，这章讲平台自己怎么部署才不宕机。核心思路：CMP 主节点 + 各机房的网关节点，主节点坏了自动切换（HA 高可用）。

两档 HA 规格　标准版：3 节点（每台 16C80G/1TB），管 5000 个资源以内；大规模版：10 节点（含独立 MySQL 集群、NFS、监控节点），管 5000–50000 个资源。客户已有 LB 和 RDS 时可以省掉对应节点。

跨 AZ 容灾　两个可用区 + 一个仲裁节点（arbiter）：某个 AZ 整体挂掉时，仲裁节点帮活着的一边凑成多数派选出新主，避免"脑裂"。前提：要有独立负载均衡（如 F5）和跨 AZ 复制的 NAS。

网络硬指标　集群间带宽 ≥100G、延迟 ≤5ms——这两个数字是跨 AZ 部署的硬门槛，达不到就别承诺。

PM 提醒：这章是你做项目资源规划的直接依据：找客户要几台什么规格的虚机、什么网络条件，照着表格开清单就行。GSB 项目的 HA 集群部署就对应这一章。
